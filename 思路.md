## 目前思路
  本系统分模拟交易和实盘两条链路并行，交易策略均按以下独立进行（比如模拟盘开仓仅排除当前模拟持仓）,每分钟轮询一次筛选新池开仓（排除当前已有持仓的）。

### 止盈止损
  调用api交易，实盘以min(1% liquidity,$50,钱包余额)投入，模拟盘按min(1% liquidity,$50)买入，以下撤仓均拿即时持仓计算比例。若结果返回失败或返回字段异常&为空，请在撤仓前至少轮换可用API重试3次以上。秒级实时监听价格：
  - 硬止盈：价格＞1.6×入盘价时，撤仓50%；＞2.1×时全部撤走；当 type 变为 completed 时全部撤走
  - 硬止损：价格＜0.7×入盘价时，撤仓50%；＜0.5×时全部撤走
  - 尘埃止损：当前持仓＜$10时，下次撤仓全部撤走
  - 风险止损：入场按以下规则检测持仓风控指标，若不满足*持仓风控指标*的阈值，则全部撤走:当前池子持仓≥$150，每4秒扫一次*持仓风控指标*；≥$100且＜$150，每8秒扫一次；≥$50且＜$100，每16秒扫一次；≥$25且＜$50，每32秒扫一次；＜$25，每64秒扫一次。若结果返回失败或返回字段异常&为空，请在撤仓前至少轮换可用API重试3次以上。
  - 取最初买入时，持仓TOP3的聪明钱，当其中任何一个聪明钱减仓比例超过25%时，撤仓50%。这项的轮询周期和*风险止损*同步，按实时持仓来。

4. 模拟交易
  考虑到免费计划的api往往用量受限，先按 x=0.2,y=2.25这组进行与实盘同步的模拟交易。软件系统提供一个x、y设置列表以供模拟和实盘交易:
  - 报价阶段：调用 Jupiter API 获取最优路径和 quote。
  - 模拟阶段：将交易打包成 Jito Bundle 格式。
  - 调用 Jito simulateBundle，检测交易是否 InstructionError；若 Tip 过低，模拟失败。
  - 记录成交后的最终代币余额。
  - 动态滑点限制：据代币当前流动性池的深度和过去几分钟的真实资产波动率进行动态计算。
  - 小费：高频轮询 Jito 提供的 tip_floor REST API，提取近期成功上链的 Bundle 的实时数据流（如 25th、50th 和 75th 百分位数）。建立一套计算公式，在网络拥堵期自动匹配至 75th 百分位数，而在平静期回落至中位数水平。保持 70% 基础优先费与 30% Jito 小费的分配比例

4. 实盘交易
  Jupiter Swap API V2，优先用 Meta-Aggregator 路径。目前考虑到成本，项目先仅从两组阈值起步。同时现在先不做contextual bandit，仅仅根据预设值进行模拟和实盘交易。但保留接口，未来考虑以过去四周的实盘&模拟交易之和作为参考，倒数第四周权重10%，第三周20%，第二周30%，最近的一周40%，然后向最优阈值组合的方向步进，进行下一周的实盘交易。实盘的同时后台同步模拟其他阈值组合。（但是现在先不用开发这个）
